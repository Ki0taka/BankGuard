
## Project Context – AML Blacklist Synchronization Platform (PFE)

## 1. Project Overview

This project is a **web-based Anti-Money Laundering (AML) blacklist management and synchronization platform** developed as a **Bachelor Final Year Project (PFE)**.

The system allows a financial institution to **manage, validate, and synchronize blacklist entries with multiple external systems**.

The primary goal is to create a **centralized platform that ensures individuals or entities identified as high risk are consistently propagated across multiple banking systems**.

This platform improves **compliance, traceability, and operational efficiency** when managing AML blacklists.

---

# 2. Core Problem the System Solves

In many financial institutions:

* Multiple systems maintain their own customer databases
* Blacklists are sometimes manually updated
* Synchronization between systems is inconsistent
* There is poor traceability of who added or modified blacklist records

This creates **serious AML compliance risks**.

The solution is to build a **central blacklist management system** that:

* Stores blacklist entries
* Tracks their origin and justification
* Synchronizes them with external systems
* Maintains complete audit history

---

# 3. System Architecture

The project is built as a **modern web application** with a **backend API**.

Backend stack:

* **Framework:** NestJS
* **Language:** TypeScript
* **Architecture:** Modular / Domain-Driven inspired
* **API style:** REST API
* **Database:** relational database (PostgreSQL recommended)
* **ORM:** Prisma or TypeORM
* **Authentication:** JWT based authentication
* **Validation:** class-validator
* **Documentation:** Swagger / OpenAPI

The backend exposes APIs that will later be consumed by a frontend interface.

# 14. Main System Workflows

## Workflow 1 – Creating a Blacklist Entry

1. User logs into the system
2. User creates a blacklist entry
3. Entry is stored with status `PENDING`
4. Supporting documents may be uploaded
5. Entry can later be approved

---

## Workflow 2 – Reviewing an Entry

1. Compliance officer reviews entry
2. Decision is recorded
3. If approved → status becomes `ACTIVE`

---

## Workflow 3 – Synchronization

1. User triggers synchronization
2. System creates a `SyncBatch`
3. For each platform:

   * a `SyncAttempt` is created
4. Platform API is called
5. Status stored (SUCCESS / FAILED)
